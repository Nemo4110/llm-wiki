# Source Snippets

> 来源类型片段库。页面仍然以 `assets/page_template.md` 为唯一主模板；本文件只提供 `## 来源笔记` 下可插入的片段。无法从来源可靠提取的字段应省略，不要为了填满模板而编造内容。

## academic_paper

用于学术论文、arXiv、会议论文、期刊文章。默认使用精简论文片段，除非用户明确要求深度论文解析。

### 来源摘要

| 维度 | 内容 |
|---|---|
| **研究背景/目标** | 该研究处于什么学术背景下？旨在解决什么核心问题？ |
| **方法** | 研究采用了什么核心方法？数据来自哪里？ |
| **结果** | 最主要的发现是什么？附关键数据或实验指标。 |
| **结论** | 研究得出了什么核心结论？其主要贡献和意义是什么？ |

### 论文四要素

| 要素 | 内容 |
|---|---|
| **根本问题** | 论文所针对的最关键矛盾、瓶颈或知识空白是什么？ |
| **切入视角** | 作者采用了什么区别于已有工作的关键洞察、新假设或新建模方式？ |
| **关键方法** | 用于验证视角并解决问题的核心机制是什么？ |
| **核心发现** | 最终得出了什么新知识或答案？用关键数据支撑。 |

### 证据与局限

- **关键证据**：列出最能支撑结论的实验、数据、图表或定性观察。
- **主要局限**：说明研究边界、未覆盖场景、实验限制或潜在偏差。
- **未来方向**：记录作者提出或可合理推出的后续研究方向。

## academic_paper_deep

用于深度论文解析。仅在用户明确要求“深度解析”、论文是页面核心来源，或该论文会支撑多个概念页时使用。

### 1. 引言

#### 1.1. 研究背景与核心问题

详细介绍本研究处于哪个宏观或微观领域，当前存在什么关键争议、挑战或现象。精准提炼本文要回答的核心研究问题。

#### 1.2. 文献综述与研究缺口

梳理作者引用的关键文献，总结现有研究的主要观点和不足。明确指出本文针对的研究缺口，即本文的创新点和必要性。

#### 1.3. 研究目标与核心假设

清晰陈述本文的研究目标。列出本文提出的核心假设或命题。若论文未明确提出假设，则归纳其隐含的核心命题。

### 2. 研究设计与方法

#### 2.1. 研究范式与方法论

解释所选用的具体研究方法及其选择原因。

#### 2.2. 数据来源与样本

说明研究数据来源、样本选取标准、规模和特征。

#### 2.3. 操作化与测量

对于系统、框架或算法类研究，说明系统核心模块、技术栈、训练/推理设置和评估指标。

### 3. 结果与发现

#### 3.1. 主要发现概述

对核心结果进行客观、中立呈现，使用具体数据支撑每一项发现。

#### 3.2. 关键数据与图表解读

选取原文中最重要的 1-3 个图或表，按以下结构解读：
- **展示内容**：该图/表展示了什么？
- **揭示关系**：揭示了怎样的关系或趋势？
- **关键数据**：提供了哪些关键数据支撑？

### 4. 讨论

#### 4.1. 结果的深度解读

解释研究发现意味着什么，以及它如何回答引言中提出的问题。

#### 4.2. 理论贡献

阐明本研究对现有理论的贡献：验证、扩展、修正某个理论，或提出新的理论框架。

#### 4.3. 实践启示

说明研究结果对相关领域实践者的具体指导意义。

#### 4.4. 局限性与未来研究

说明本研究局限，并基于发现和局限提出后续研究方向。

### 5. 结论

凝练总结全文，再次强调最重要的发现和贡献。

### 最终双重总结

- **一句话总结（核心价值）**：将问题、视角、方法和发现熔合成一个单一、连贯、凝练的句子。
- **一句话总结（大白话版）**：用非专业读者也能理解的比喻概括核心观点，避免术语堆叠。

## technical_article

用于技术博客、工程文章、教程、经验总结。

### 来源摘要

- **背景场景**：作者讨论的工程场景、业务约束或技术背景是什么？
- **核心观点**：文章最重要的主张或判断是什么？
- **实现方法**：如果文章包含方案，记录关键步骤、架构或代码模式。
- **实践建议**：哪些建议可被迁移到其他项目？

### 可迁移洞察

- **适用场景**：该经验在什么条件下成立？
- **反模式**：作者避免或反对的做法是什么？
- **工程权衡**：该方案牺牲了什么，换来了什么？

## code_repository

用于 GitHub 仓库、开源项目、代码包、示例工程。

### 来源摘要

- **项目目标**：这个项目解决什么问题？
- **核心抽象**：最重要的模块、类、协议或数据结构是什么？
- **架构组成**：系统如何分层、如何通信、如何扩展？
- **关键依赖**：依赖哪些框架、服务或运行时？

### 架构洞察

- **可复用模式**：哪些设计可以迁移到其他项目？
- **设计权衡**：项目在复杂度、性能、可维护性、可移植性之间如何取舍？
- **类似项目对比**：它与已知方案相比有什么差异？

## note

用于个人笔记、想法、会议记录、临时观察。

### 来源摘要

- **原始问题**：这条笔记试图回答或暴露什么问题？
- **初步想法**：作者当前的判断、猜测或思路是什么？
- **待验证假设**：哪些内容需要后续证据确认？
- **后续行动**：下一步需要阅读、实验、讨论或实现什么？

## documentation

用于官方文档、规范、API 文档、产品说明。

### 来源摘要

- **适用对象**：文档面向哪些用户、系统或使用场景？
- **核心能力**：文档定义了哪些能力、接口或约束？
- **使用约束**：版本、权限、配额、兼容性或安全边界是什么？
- **关键接口**：记录核心命令、API、配置项或协议。

### 操作知识

- **最小可用路径**：完成一个基础任务的最短步骤是什么？
- **常见错误**：哪些误用会导致失败或隐患？
- **版本/兼容性注意事项**：哪些行为会随版本变化？

## book

用于书籍、章节、长篇课程、系统性讲义。

### 来源摘要

- **主题范围**：本书或章节覆盖什么问题域？
- **核心论点**：作者最重要的主张是什么？
- **概念体系**：作者如何组织概念、层级和推理路径？
- **关键案例**：哪些例子最能说明其观点？

### 知识迁移

- **可复用框架**：哪些框架可抽象为 wiki 中的概念页？
- **与已有知识关系**：它补充、修正或挑战了哪些已有页面？
- **阅读后的开放问题**：还有哪些问题需要更多来源支撑？

## other

用于无法归类但仍有知识价值的来源。

### 来源摘要

- **来源类型**：这是什么材料？为什么值得 ingest？
- **核心内容**：它提供了什么事实、观点、案例或问题？
- **可沉淀知识**：哪些内容应进入知识卡片？
- **不确定性**：哪些结论需要额外来源验证？
