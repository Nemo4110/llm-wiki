# Ingest 规则

> 指导 Agent 如何处理不同类型的资料，生成专业、严谨、深度的知识内容。

## sources/ 完整性红线

在启动任何 ingest 流程之前，必须确认以下前提：

1. **来源真实性**：`sources/` 中的文件必须是用户提供的原始文件，或 Agent 通过真实网络请求（curl、playwright 等）获取的文件。
2. **禁止伪造**：绝对禁止将 LLM 生成的内容、摘要、推测写入 `sources/` 并伪装成原始资料。
3. **失败即停止**：如果用户提供了 URL 但获取失败（404、403、付费墙、反爬），不得在 `sources/` 创建替代文件。应标记为 pending 并告知用户。
4. **前置检查**：
   - 文件已在 `sources/` → 直接读取 ingest
   - 用户提供了 URL/DOI → 先工具获取，成功后再 ingest
   - 用户只给了标题/作者，无 URL，文件不在 `sources/` → Agent 应**主动使用网络搜索工具**查找开放获取版本，反馈给用户确认后再下载；搜索无果则标记 pending 并告知用户

## 核心写作原则

### 专家视角与第一性原理思维

**必须遵循的原则**：

1. **第一性原理思维**：
   - 从最基本的原理（物理/数学/经济/心理学）出发
   - 推导概念的本质，而非表面现象
   - 建立因果关系，而非相关性描述

2. **专家角色定位**：
   - 以该领域资深学者/高级工程师的视角撰写
   - 展现深度理解，而非浅层知识罗列
   - 使用专业术语，同时提供精准定义

3. **辩证思维**：
   - 分析优势与局限
   - 比较不同理论的异同
   - 批判性地评价，而非简单接受

4. **严谨性要求**：
   - 提供数学公式和推导（如适用）
   - 引用实验数据和统计显著性
   - 明确假设前提和适用边界

### 写作风格指南

#### ✅ 应该做的

- **深度分析**：解释"为什么"，而不仅仅是"是什么"
- **建立联系**：连接到已有理论、实践、其他概念
- **提供证据**：实验数据、数学证明、实证案例
- **批判思考**：分析局限、挑战、争议点
- **前瞻视角**：讨论未来方向、开放问题

#### ❌ 避免的做法

- **浅层描述**：仅列出要点，缺乏深度分析
- **简单总结**：机械概括，缺少专家洞见
- **缺乏批判**：全盘接受，不分析局限
- **孤立知识**：不建立与其他概念的联系
- **模糊表述**：使用"可能"、"大概"等不确定表达

## 通用原则

1. **提取而非复制**：不要大段复制原文，要提取结构化洞察，并深度分析
2. **连接而非孤立**：每个新概念都要链接到现有概念，建立知识网络
3. **溯源**：每个声明都要能追溯到具体来源
4. **专家视角**：以该领域专家的身份，提供深度见解和批判性分析
5. **知识页优先**：wiki 页面默认是知识页，不是来源页；来源材料只通过「来源笔记」进入页面
6. **不为模板造内容**：无法从来源可靠提取的字段应省略，不要写空泛占位、通用废话或推测性补全

## 单页双层模型

llm-wiki 采用“逻辑双层、物理单页”：

1. **知识层**：页面主体沉淀可复用概念、机制、边界、判断和知识网络关系
2. **来源层**：`## 来源笔记` 记录材料证据、原始主张、关键数据和局限

Agent 仍然只维护一个 `wiki/*.md` 页面。不要为了区分来源和知识而额外创建“来源页 + 记录页”两套文档，除非用户明确要求。

## Ingest 三阶段决策

### 1. 先判断目标知识页

优先回答：这份材料最终应该沉淀到哪个知识页面？

- 如果材料引入新概念、方法、系统或问题 → 创建对应知识页
- 如果材料补充已有概念 → 更新已有页面
- 如果材料只是某个概念的案例或证据 → 追加到已有页面的「来源笔记」或相关段落
- 如果材料是未成熟想法 → 创建或更新 draft 页面

### 2. 再判断来源类型

来源类型只决定 `## 来源笔记` 下插入哪个片段，片段库见 `assets/source_snippets.md`：

- `academic_paper`：学术论文、arXiv、会议论文、期刊文章
- `technical_article`：技术博客、工程文章、教程、经验总结
- `code_repository`：GitHub 仓库、开源项目、代码包
- `note`：个人笔记、想法、会议记录
- `documentation`：官方文档、规范、API 文档、产品说明
- `book`：书籍、章节、长篇课程、系统性讲义
- `other`：无法归类但有知识价值的来源

### 3. 最后判断详略程度

默认使用精简片段。只有满足以下条件之一，才展开 `academic_paper_deep` 或类似深度结构：

- 用户明确要求“深度解析这篇论文”或“按论文结构完整分析”
- 该论文是页面的核心来源，而不是某个概念的旁证
- 该论文会支撑多个概念页、方法页或综述页
- 论文的方法、实验或理论贡献需要被长期复用

## 按资料类型的处理策略

### 学术论文

**提取内容**：

- 研究问题
- 方法创新
- 关键结果
- 局限性和未来方向
- 可迁移到知识层的机制、模型或边界条件

**创建页面**：

- 优先创建或更新概念页、方法页、系统页、问题页
- 只有当论文自身是知识网络中的核心对象时，才创建论文同名页面
- 如果方法是通用贡献，创建或更新方法页面
- 更新相关领域综述页面

### 技术博客/文章

**提取内容**：

- 核心观点
- 代码模式（如果有）
- 实践经验/教训
- 工程权衡和反模式

**创建页面**：

- 技术主题页面
- 模式库（如果是实现技巧）
- 更新相关实践或架构页面

### 代码仓库

**提取内容**：

- 架构设计
- 关键抽象
- 依赖关系
- 可复用设计模式和边界条件

**创建页面**：

- 项目页面（如果项目本身值得长期跟踪）
- 设计模式页面（提取通用模式）
- 更新相关技术栈或系统架构页面

### 笔记/想法

**提取内容**：

- 原始洞察
- 待验证的假设
- 问题/待办
- 与已有知识的关系

**创建页面**：

- 草稿页面（标记为 draft 状态）
- 或追加到相关现有页面

### 官方文档/规范

**提取内容**：

- 核心能力
- 接口、命令或配置项
- 版本、权限、兼容性和安全边界
- 最小可用路径和常见误用

**创建页面**：

- 技术能力页面
- API/配置页面
- 更新相关实践页面

## 页面创建决策树

```text
材料是否提供可复用知识？
  ├─ 否 → 不 ingest，或记录为 pending/待验证
  └─ 是 → 是否已有承载该知识的页面？
           ├─ 是 → 更新现有页面，并在「来源笔记」追加来源片段
           └─ 否 → 创建新知识页，使用 page_template.md 作为主模板
                    └─ 根据来源类型从 source_snippets.md 插入来源片段
```

## 合并策略

当新资料与现有页面相关时：

1. **补充**：添加新信息到合适段落
2. **对比**：如果新资料有不同观点，添加「不同视角」段落
3. **更新**：如果新资料是更新的版本，替换旧信息并注明
4. **冲突标记**：如果存在矛盾，保留两者并标记 `⚠️ 矛盾：来源X说...，来源Y说...`
