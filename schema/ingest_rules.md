> 完整写作深度指南见 `assets/ingest_rules.md`。

## 红线规则（执行 ingest 前必须检查）

1. **文件是否在 `sources/` 中？**
   - 是 → 继续
   - 否 → 用户是否提供了可获取的 URL/DOI？
     - 是 → 执行网络获取（curl / playwright）→ 成功则继续，失败则标记 pending
     - 否 → Agent **主动使用网络搜索工具**查找开放获取版本 → 反馈给用户确认 → 确认后下载获取；搜索无果则标记 pending

2. **是否打算将 LLM 生成内容写入 `sources/`？**
   - **绝对禁止**。`sources/` 只存放原始资料。

3. **获取失败时如何处理？**
   - 不在 `sources/` 创建任何文件
   - 在 `wiki/` 创建 `[[Pending: 来源名]]` stub 页面
   - 在 `log.md` 记录失败原因
   - 回复用户说明情况

## 按资料类型的处理策略

| 类型 | 提取重点 | 页面创建 |
|------|---------|---------|
| 学术论文 | 研究问题、方法创新、关键结果、局限性 | 论文同名页面 + 方法页面 |
| 技术博客 | 核心观点、代码模式、实践经验 | 技术主题页面 |
| 代码仓库 | 架构设计、关键抽象、依赖关系 | 项目页面 + 设计模式页面 |
| 笔记/想法 | 原始洞察、待验证假设、问题 | 草稿页面（draft）或追加现有页 |

## 页面创建决策树

```
新概念？
  ├─ 是 → 是否已存在相关页面？
  │        ├─ 是 → 更新现有页面，添加链接
  │        └─ 否 → 创建新页面（含 frontmatter + 一句话定义）
  └─ 否 → 更新现有页面
```

## 合并策略

当新资料与现有页面相关时：

1. **补充**：添加新信息到合适段落
2. **对比**：如果新资料有不同观点，添加「不同视角」段落
3. **更新**：如果新资料是更新的版本，替换旧信息并注明
4. **冲突标记**：如果存在矛盾，保留两者并标记 `⚠️ 矛盾：来源X说...，来源Y说...`
